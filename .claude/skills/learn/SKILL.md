# Learn - 用語学習・記録スキル

知らない用語や概念を対話で学び、`~/repos/kubutaku-memory/vault/wiki/pages/` に Obsidian ページとして記録する。

## 保存先

- ページ: `~/repos/kubutaku-memory/vault/wiki/pages/{kebab-case}.md`
- 目次: `~/repos/kubutaku-memory/vault/wiki/index.md`
- ログ: `~/repos/kubutaku-memory/vault/wiki/log.md`

## ワークフロー

### Step 1: ヒアリング

ユーザーに以下を確認する:

1. **何について学びたいか** — 用語・概念名
2. **なぜ学びたいか** — 背景や動機（業務で出てきた、記事で見かけた、など）

ユーザーが `/learn ○○` のように用語を添えて呼び出した場合は、用語名の確認は不要。背景だけ聞く。

### Step 2: 重複チェック

`~/repos/kubutaku-memory/vault/wiki/pages/` に同じ用語のファイルが既にないか確認する。既にある場合はユーザーに伝え、更新するか別の用語にするか確認する。

### Step 3: 対話で学ぶ

- まずユーザーが現時点で知っていることを聞く
- 概念をわかりやすく説明し、対話しながら理解を深める
- ユーザーの実務（データエンジニアリング、BigQuery、dbt 等）に紐づけて具体例を出す
- ユーザーの疑問に答え、理解が十分になるまで対話を続ける

### Step 4: 統括を書いてもらう

- 「自分の言葉でまとめてみてください」と促す
- ユーザーがまとめた内容をレビューし、誤解があれば指摘する
- OKになるまで繰り返す

### Step 5: 調査・ページ作成

ユーザーの統括が固まったら、以下を行う:

1. **WebSearch で複数ソースを調査する**
   - 1st: 公式ドキュメントから定義を取得
   - 2nd: 別の記事・資料から定義を取得
   - 3rd: さらに別の視点からの定義を取得
   - 各定義の根拠となるURLを記録する

2. **タグを自動判別する**（複数可）

   | カテゴリ | 例 |
   |---------|-----|
   | 種別 | 概念、SaaS、OSS、サービス、手法、パターン |
   | ベンダー | Google、Amazon、Microsoft、Snowflake |
   | ドメイン | データエンジニアリング、データモデリング、インフラ |

3. **ページ作成前に既存ページのスタイルを確認する** — `vault/wiki/pages/` に既存ページが 1 つ以上ある場合、**必ず 1 つを読んでスタイル（見出しレベル・段落 vs 箇条書き・強調の使い方など）を確認し、それに揃える**。テンプレートだけを見て作ると既存ページと乖離することがある。

4. **vault の CLAUDE.md を確認する** — `~/repos/kubutaku-memory/CLAUDE.md` に vault 固有の運用ルール（Zettelkasten 方式など）が書かれているので従うこと。

5. **ページを作成する** — 以下のフォーマットで `vault/wiki/pages/{kebab-case}.md` に保存

```markdown
---
tags: [自動判別したタグ]
---

作成日：YYYY-MM-DD

## 統括

（Step 4 でユーザーがまとめた内容）

## 定義

**1st（公式ドキュメント等の出典名）:**
（公式ドキュメントからの定義）

**2nd（別の記事・資料の出典名）:**
（別の記事・資料からの定義）

**3rd（さらに別の視点の出典名）:**
（さらに別の視点からの定義）

## 参考資料

- [1st の記事タイトル](URL)
- [2nd の記事タイトル](URL)
- [3rd の記事タイトル](URL)
```

6. **index.md にエントリを追加**: `- [[ページ名]] — 一行の説明`
7. **log.md に記録**: `## [YYYY-MM-DD] learn | 用語名`
8. **本文中で別概念に触れたら `[[wikilink]]` を張る** — Zettelkasten 方式に従い、関連ページが未作成であっても stub として wikilink を張る（将来の学習対象リストになる）

#### Markdown スタイルガイド

ページ作成時は以下のルールに従い、Obsidian で読みやすいフォーマットにする。ただし既存ページとスタイルが異なる場合は**既存ページを優先**する。

- **見出し**: セクション見出しは `##` を使う（h1 相当のタイトルは frontmatter の tags で代替）。見出しの前後に空行を入れる
- **段落**: 段落間は空行1行で区切る。同一段落内で改行しない
- **リスト**: リスト項目間に空行は入れない。ネストは半角スペース2つ。リストの前後に空行を入れる
- **太字・強調**: `**太字**` は見出し代わりに使わない。文中のキーワード強調に限定する
- **コードブロック**: インラインは `` `コマンド名` ``、複数行は ` ``` ` で囲み言語指定する
- **テーブル**: ヘッダ行とセパレータ行を必ず含める。セル内は簡潔にする
- **定義セクション**: 各定義（1st/2nd/3rd）は空行で区切り、冒頭に `**1st（出典名）:**` の形式でラベルを付ける。本文は段落で記述する（箇条書きにしない）
- **リンク**: URL は必ず `[タイトル](URL)` 形式にする。生 URL を貼らない
- **wikilink**: 本文中で別概念に触れたら `[[ページ名]]` を張る。未作成ページへのリンクでも stub として有効

#### 品質基準

- **1ファイル = 1つの用語・概念**（Zettelkasten の「アトミックノート」原則）
- **日本語で記述**する
- **作成日**は `date +%Y-%m-%d` で取得する
- **統括**はユーザー自身の言葉をそのまま使う（勝手に書き換えない）
- **定義の1st/2nd/3rd**はそれぞれ異なるソースから取得する（セカンドオピニオン、サードオピニオン）
- **定義は日本語で記述**する。英語ソースの場合は日本語に翻訳する（固有名詞・コマンド名・技術用語はそのまま残してよい）
- **固有名詞は正式名称（英語原綴りなど）を使う**。カタカナ転写に置き換えない（例: ❌ ツェッテルカステン／ニクラス・ルーマン → ⭕ Zettelkasten／Niklas Luhmann）。公式ドキュメントや Wikipedia で正式表記を確認すること
- **テンプレート外のセクションを勝手に追加しない**（例: 「使い分けチートシート」「比較表」「所感」などは作らない）。必要なら別の独立したページとして切り出す
- **参考資料**は各定義と1対1対応させる

### Step 6: ユーザー確認

作成したファイルの内容をユーザーに見せ、修正点がないか確認する。修正があれば反映する。修正後は改めて全体を読み直し、スタイルガイドと既存ページスタイルの両方に合致しているか確認する。

ユーザーが OK したら完了。`~/repos/kubutaku-memory` は現時点で git 管理されていないため、コミット・プッシュは行わない。
