# Marp Design Review

Marp の Markdown プレゼン資料を HTML に変換し、Playwright CLI でスクリーンショットを撮って視覚的にデザインをチェックして、ガイドライン違反を自動修正するスキル。

## なぜこのスキルが必要か

Marp はテキストのオーバーフロー自動制御を持たない。スライドに情報を詰め込みすぎると、テキストやテーブルがスライド外にはみ出すが、Markdown 編集中はそれに気づけない。HTML に変換してブラウザで確認して初めて視覚的な問題が判明するため、「変換 → スクリーンショット → 視覚チェック → 修正」のサイクルを自動化する。

## ワークフロー

```
Step 1: HTML 変換
Step 2: Playwright CLI でスクリーンショット撮影・視覚チェック
Step 3: レビュー結果の報告
Step 4: 修正（structured-workflow で plan.md → 修正実行）
Step 5: 再変換・再チェック（問題が解消されるまで繰り返す）
```

### Step 0: テーマの確認

対象の `.md` ファイルの frontmatter を確認し、カスタムテーマが設定されているかチェックする。

- カスタムテーマがない場合: プロジェクトに `theme.css` が存在するか確認する。なければ後述の「カスタムテーマ CSS」セクションを参考に作成を提案する
- カスタムテーマがある場合: そのテーマの内容を読み込み、利用可能なクラスを把握する

### Step 1: HTML 変換

対象の `.md` ファイルを HTML に変換する。

```bash
npx @marp-team/marp-cli <input.md> -o <output.html>
```

出力先は `.claude/tmp/YYYYMMDD-marp-review/` に保存する（`date +%Y%m%d` で日付取得）。

**重要: リポジトリルートから実行すること。** `.marprc.yml` がルートにある場合、そこに `engine`、`theme`、`themeSet`、`allowLocalFiles` 等の設定が含まれている。ルート以外から実行すると、カスタムエンジン（markdown-it-container 等のプラグイン）が読み込まれず、`:::` コンテナ構文がレンダリングされない。

```bash
# 正しい: リポジトリルートから実行
cd <リポジトリルート> && npx @marp-team/marp-cli <相対パス/input.md> -o .claude/tmp/YYYYMMDD-marp-review/preview.html

# 間違い: .md のディレクトリから実行（.marprc.yml が見つからない）
cd <input.md のディレクトリ> && npx @marp-team/marp-cli ...
```

`.marprc.yml` がない場合は、テーマ指定等のオプションを明示的に渡す:

```bash
npx @marp-team/marp-cli <input.md> --allow-local-files --theme <theme.css> -o <output.html>
```

### Step 2: スクリーンショット撮影・視覚チェック

Playwright CLI でブラウザを開き、各スライドのスクリーンショットを撮影して視覚チェックする。

#### ブラウザの起動と HTML の読み込み

```bash
playwright-cli open
playwright-cli goto file://<output.html の絶対パス>
```

#### スライドのナビゲーションとスクリーンショット

Marp の HTML 出力は URL ハッシュ（`#1`, `#2`, ...）でスライドを移動できる。

```bash
# スライド 1（デフォルト表示）のスクリーンショット
playwright-cli screenshot --filename=<output-dir>/slide-01.png

# スライド 2 に移動してスクリーンショット
playwright-cli goto file://<output.html の絶対パス>#2
playwright-cli screenshot --filename=<output-dir>/slide-02.png

# スライド 3 に移動してスクリーンショット
playwright-cli goto file://<output.html の絶対パス>#3
playwright-cli screenshot --filename=<output-dir>/slide-03.png

# ... 以降、全スライドについて繰り返す
```

**効率化のコツ:**
- まず `.md` ファイルのスライド区切り（`---`）を数えて総スライド数を把握する
- 全スライドのスクリーンショットを撮り終えてから、まとめて Read ツールで読み込む
- 10枚ずつ並列で Read ツールに渡すと効率的

#### 撮影後のブラウザ終了

```bash
playwright-cli close
```

#### 視覚チェック

撮影した画像を Read ツールで読み込み、以下のデザインガイドラインに照らしてチェックする。**すべてのスライドを確認すること。**

チェック観点:
1. **はみ出し**: テキスト、テーブル、画像がスライド領域からはみ出していないか
2. **情報量**: 1スライドに詰め込みすぎていないか
3. **余白バランス**: 上下左右に十分な余白があるか、偏りがないか
4. **見出し階層**: 不適切な見出しレベルが使われていないか
5. **テーブルサイズ**: テーブルが大きすぎてスライドに収まっていないか
6. **可読性**: フォントが小さすぎないか、コントラストは十分か
7. **画像の表示**: 画像が壊れた表示（alt テキストだけ表示）になっていないか
8. **見出しの統一性**: スライド間で見出しレベルが統一されているか（h1 と h2 の混在など）

**未作成スライド（タイトルだけで中身が空）はデザインレビューの対象外。** レビュー結果で「未作成」と記載するだけでよい。

### Step 3: レビュー結果の報告

問題のあるスライドを一覧で報告する。

```
## レビュー結果

### スライド 3: [スライドタイトル]
- ❌ テキストが下部からはみ出している
- ⚠️ 箇条書きが8項目あり、情報量が多い

### スライド 7: [スライドタイトル]
- ❌ テーブルが右側にはみ出している

### 問題なし: スライド 1, 2, 4, 5, 6, 8
```

問題がなければその旨を報告して終了する。

### Step 4: 修正

問題がある場合、structured-workflow のフローに沿って修正する。

1. レビュー結果を `.claude/tmp/YYYYMMDD-marp-review/plan.md` に落とし込む
2. plan.md にはスライドごとの修正方針を記載する（例: 「スライド3を2枚に分割」「テーブルの列を絞る」）
3. ユーザーに plan.md を提示し、アノテーションサイクルで修正方針を確定する
4. 承認後、`.md` ファイルを修正する

**厳守: ユーザーが書いたコンテンツの文章を勝手に書き換えない。** はみ出しや情報量の問題は、CSS クラス指定（`small`, `compact` 等）やスライド分割で対応する。文章の短縮・言い換え・要約による修正は禁止。添削の提案は別途行ってよいが、勝手に適用しない。

### Step 5: 再チェック

修正後、再度 HTML 変換 → Playwright CLI スクリーンショット → 視覚チェックを行い、問題が解消されたか確認する。問題が残っていれば Step 3 に戻る。

## デザインガイドライン

### 1スライドの情報量

- テキスト量: **100〜150文字以内**（日本語）を目安にする
- 箇条書き: **3〜5項目が理想、最大7項目**。超える場合はスライドを分割する
- ネストの深さ: **2階層まで**。3階層以上のネストは情報の整理が不十分なサイン
- 原則「**1スライド1メッセージ**」。複数の論点を1枚に詰め込まない

### 見出し

- **h1**: スライドタイトル（各スライドに1つ）
- **h2**: サブタイトル・セクション見出し
- **h3**: 小見出し（補足的な区分）
- **h4〜h6 は使わない**: フォントサイズが本文とほぼ同じになり、視覚的に区別がつかない。h4 以降を使っている場合は、スライド分割や構成見直しで対応する
- タイトルが長い場合は `<!-- fit -->` でフィッティングを検討する

### テーブル

- **列数**: 5列以下を推奨。超える場合は重要な列だけに絞る
- **行数**: 5行以下を推奨。超える場合はスライドを分割する
- テーブルの内容が多い場合の対処法（優先順で）:
  1. 不要な列・行を削る
  2. スライドを分割する
  3. テーブル用に `<style>table { font-size: 0.7em; }</style>` でフォントを縮小する（最終手段）

### コードブロック

- **15〜20行以内** にする
- 長いコードは重要部分だけ抜粋し、全体は別途共有する
- コードブロックには Marp の auto-scaling 機能があるため、多少の超過は許容される

### 画像

- **サイズは必ず明示指定する**。指定がない画像はレビューで指摘する
- 用途別のサイズ目安:
  - **補助的な画像**（プロフィール写真、アイコン等）: `w:200`〜`w:300`
  - **スクリーンショット・例示画像**: `w:500`〜`w:900`（内容の読みやすさに応じて調整）
  - **図解・ダイアグラム**: `w:600`〜`w:800`（スライド幅の50〜70%程度）
- **配置はスライドの主役が何かで決める**。デフォルト配置をそのまま使わず、意図を持って選択する:
  - 「このスライドで聴衆に一番見てほしいのは画像かテキストか？」を判断基準にする
  - 画像がメインコンテンツ → 中央配置
  - 画像がテキストの補足 → カラムレイアウトや `bg right` でテキストの横に配置
- 配置パターン:
  - **テキストと並べる場合**: カラムレイアウト内に `![w:250](image.jpg)` で配置するか、`![bg right:30%](image.jpg)` で背景配置
  - **メイン画像を中央に見せる場合**: `![w:700](image.jpg)` + `<style scoped>p:has(img) { text-align: center; }</style>` で中央寄せ。Marpのインライン画像はデフォルト左寄せなので、中央配置には明示的なスタイルが必要
  - **画像だけのスライド**: `![bg contain](image.jpg)` で背景配置。ただしテキストがあると被るので、テキストなしのスライドでのみ使用する
- `![bg contain]` はテキストと併用しない。画像が背景全体に広がり、テキストが上に重なって読めなくなる
- 画像がスライドの内容を圧迫していないか確認する。テキストが画像に押されて窮屈になっている場合は、画像サイズを縮小するかスライドを分割する
- **配置を変更したら必ずスクリーンショットで確認する**。Marpの画像配置は直感と異なる挙動をすることが多いため、目視確認なしで配置を決めない

### 余白・レイアウト

- スライドの四辺に十分な余白があること（テーマのデフォルト padding を尊重する）
- テキストやテーブルがスライドの端ギリギリまで来ていたら情報量を減らす
- 上半分だけ、下半分だけに偏っていないか確認する
- **要素間の余白が十分か確認する**。テキストとパネル、見出しと本文、リストとカードなどの間が詰まりすぎていると窮屈に見える。「窮屈に感じないか」を必ずチェックする
- 文字間隔（tracking）が適切か確認する。大きな見出しが間延びしている場合は `.tracking-tight`、小さいラベルやキャプションが詰まっている場合は `.tracking-wide` の適用を検討する

### 可読性

- 本文フォントサイズが極端に小さくなっていないか（テーマのデフォルト以下になっていないか）
- 背景色と文字色のコントラストが十分か
- 太字・色分け等の強調は1スライドに2〜3箇所まで。多用すると逆に読みにくくなる

### 階層・構造の表現

**原則: CSS疑似要素による接続線（`::before`, `::after`）で図を描こうとしない。** Marp のスライドレンダリングでは `position: absolute` の座標計算が意図通りにならず、線が途切れたり位置がずれたりする。結果として「何を表現しているかわからない」成果物になる。

**なぜこの失敗が起きるか:**
- 「組織図っぽく見せたい」という装飾発想が先行し、「聴衆が一目で構造を理解できるか」の検証を怠る
- CSS 接続線は Web ページでは機能しても、Marp のスライド変換（HTML → PDF/画像）では座標がずれやすい
- 見た目の「かっこよさ」を追求して複雑にした結果、壊れたときに何も伝わらなくなる

**階層を伝えるために使うべき手法（優先順）:**
1. **2カラム + ボーダー付きパネル**: カテゴリ名を `panel-border` で強調し、子要素をリストで並べる。最もシンプルで確実
2. **インデント + 色の差**: 親を太字・大きめ、子を小さめ・別色で視覚的に階層を区別する
3. **テーブル**: 2列テーブル（カテゴリ / 手法）でフラットに整理する

**使ってはいけない手法:**
- CSS `::before` / `::after` で接続線を描く
- SVG をインラインで埋め込んで図を描く
- `position: absolute` に依存するレイアウト

## スタイル設定方式

よく使うパターンは**カスタムテーマ CSS の class** で定義し、一時的な微調整は **Marp ディレクティブ**で対応する。

### カスタムテーマ CSS

プロジェクトのルートに `theme.css` を作成し、md ファイルの frontmatter で指定する。

```yaml
---
marp: true
theme: custom-theme
paginate: true
---
```

テーマ CSS のベーステンプレート:

```css
/* @theme custom-theme */
@import 'default';

/* --- フォントサイズ調整クラス --- */

/* 情報量が多いスライド向け（テーブル、詳細説明等） */
section.small {
  font-size: 0.8em;
}

/* 控えめに縮小（少しだけ収まらない時） */
section.compact {
  font-size: 0.9em;
}

/* タイトルスライド向け（中央寄せ・大きめ） */
section.title {
  display: flex;
  flex-direction: column;
  justify-content: center;
  align-items: center;
  text-align: center;
}

/* --- 文字色クラス --- */

/* 強調（赤系） */
section.accent {
  --color-highlight: #e74c3c;
}
section.accent strong {
  color: var(--color-highlight);
}

/* 落ち着いたトーン（グレー系、補足スライド向け） */
section.muted {
  color: #555;
}

/* --- テーブル専用 --- */

/* テーブルのフォントだけ縮小 */
section.small-table table {
  font-size: 0.7em;
}
```

md ファイルでの使い方:

```markdown
---
<!-- _class: title -->
# プレゼンタイトル

---
<!-- _class: small -->
# テーブルが多いスライド
| 列1 | 列2 | 列3 |
|-----|-----|-----|

---
<!-- _class: accent -->
# 重要なポイント
**ここが強調される**
```

### Marp ディレクティブ（一時的な微調整）

スライドごとに個別でスタイルを変えたい場合、ディレクティブを使う。カスタムテーマに class を増やすほどでもない一回限りの調整に向いている。

```markdown
---
<!-- _color: #333 -->
<!-- _backgroundColor: #f5f5f5 -->
# このスライドだけ背景をグレーに
```

主なディレクティブ:
- `<!-- _color: ... -->` — 文字色
- `<!-- _backgroundColor: ... -->` — 背景色
- `<!-- _class: ... -->` — テーマの CSS クラスを適用
- `<!-- _header: ... -->` — ヘッダーテキスト
- `<!-- _footer: ... -->` — フッターテキスト
- `<!-- _paginate: false -->` — ページ番号を非表示

### 修正時のスタイル適用方針

Step 4 の修正で、スタイル調整が必要な場合は以下の優先順で対応する:

1. **カスタムテーマの class を適用する**（`small`, `compact`, `small-table` 等）
2. **スライドを分割する**（情報量が多すぎる場合）
3. **Marp ディレクティブで個別調整する**（そのスライド固有の色・背景等）
4. **カスタムテーマに新しい class を追加する**（同じパターンが3回以上出現する場合）

**ユーザーの文章を短縮・書き換えて収める方法は使わない。**
