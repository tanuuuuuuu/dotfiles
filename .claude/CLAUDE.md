## 開発原則

- KISS: 最もシンプルな解決策を選ぶ。複雑さは必要になったときに追加する
- YAGNI: 現時点で必要な機能だけを実装する。「将来使うかも」で作らない
- DRY: 同じロジックが3箇所以上に現れたら共通化を検討する（2箇所は許容）
- SOLID: クラスやモジュールの設計時に適用する（特に単一責任の原則を重視）
- MECE: 選択肢の提示や分析では、漏れなくダブりなく整理する

## ワークフロー

### 設計フェーズ

- 複数案をメリット・デメリット付きで MECE に比較し、推奨案を1つ選ぶ
- 推奨案が将来どういう変更で破綻するかも提示する

### 実装フェーズ

- 段階的に進める: 小さな変更を積み重ね、複数タスクを同時並行で進めない
- エラーは根本原因を特定してから対処し、解決してから次へ進む
- 周囲の似た実装を探し、それを参考にする

### 並行開発（git worktree）

- 独立した複数タスク・複数ブランチの同時進行になりそうだと察知したら、明示的な依頼がなくても git worktree での並行開発を提案する
  - 例: 別ブランチの作業を頼まれた、レビュー待ちの間に別機能に着手したい、モノレポで複数プロジェクトを同時に触る
- worktree は `~/repos/<リポジトリ名>-<ブランチ名>` に作成し、各 worktree で別々の Claude Code セッションを起動する
- 作業完了後は `git worktree remove` での削除まで案内する

### 調査フェーズ

- 調査プロセスをリアルタイムに報告する
- 結論をまとめる時は、重要な証拠も一緒に提示する

## 環境設定

- 設定ファイルは `~/dotfiles` リポジトリで一元管理している
- 新しい設定ファイルを追加・変更する場合は dotfiles リポジトリを更新する

### Bash コマンド

- ファイル削除は `rm` ではなく `trash` コマンドを使用（復元可能にするため）
- ファイル名・パス検索は `find` ではなく `fd` コマンドを使用（高速・`.gitignore` 尊重・構文が簡潔）
  - 例: `fd 'pattern' path/ --type f --exclude node_modules`
  - `find` でしか実現できない用途（`-newer`、複雑なパーミッション条件など）の場合のみ `find` を使う
- ファイル内容検索は `grep` ではなく `rg`（ripgrep）コマンドを使用（高速・`.gitignore` 尊重・デフォルトで再帰検索）
  - 例: `rg 'pattern' path/ --type py`
  - `grep` でしか実現できない用途や、パイプ経由でテキスト処理する場合のみ `grep` を使う
- 過去の会話履歴・エージェントセッションは `ctx` で検索できる（指示がなくても、関連しそうなら行動前に参照する）。詳細は skill `ctx-agent-history-search`
- 「最近」「直近」など現在日時が必要な場合は、必ず `date` コマンドで取得
- `brew install` 実行後は `~/dotfiles/Brewfile` にも追加する
  - 通常パッケージ: `brew "パッケージ名"`
  - cask: `cask "パッケージ名"`
  - tap が必要な場合: `tap "タップ名"` も追加
